# NewZealand 旅行路书 App 设计规范

这份文档是项目后续设计和实现的硬性参考。任何页面、组件、动效、图片和内容组织都必须优先遵守这里的方向。

## 1. 产品定位

NewZealand 是一个给 2026 国庆新西兰同行人使用的手机端旅行路书 App。

它不是公开发布的旅游攻略站，也不是把 Markdown 原文渲染成网页。它的目标是把完整路书转换成一个好看、顺手、能在旅途中真实使用的移动端体验。

核心使用场景：

- 出发前：查看整体路线、每日安排、待预订事项、证件和携带清单。
- 讨论时：用图片、地点卡片、选项卡帮助同行人做选择。
- 旅途中：快速知道今天几点去哪、开多久、住哪里、哪些事项必须完成。
- 临时调整时：能清楚看到备选方案、风险提示和当天重点。

设计优先级：

1. 好看，有 2026 年移动端审美。
2. 信息清楚，不能为了炫技牺牲可读性。
3. 交互轻松，有完成感和推进感。
4. 手机优先，桌面只是兼容。
5. 内容基于路书，但表现方式以前端组件为主。

## 2. 总体视觉方向

主方向命名：Duo Trip + Soft Planner + Liquid Lake。

视觉参考组合：

- Duolingo：借鉴游戏化路径、大圆角按钮、强反馈、任务完成感、彩色状态。
- Finch：借鉴柔和卡片、每日任务、checklist、温暖但不幼稚的情绪。
- Tiimo / Structured：借鉴时间块、日程规划、清晰的每日时间线。
- iOS 26 Liquid Glass：借鉴半透明浮动导航、玻璃质感、大圆角、轻盈层次。
- 新西兰自然风景：作为图片和配色的主题来源，包括湖水、雪山、草地、峡湾、日落。

最终效果应该像一个精致的旅行闯关手册：

- 每一天像一关。
- 每个地点像一个可解锁节点。
- 每个 To-do 像可以完成的任务。
- 每个景点选择像可以投票讨论的漂亮卡片。
- 交通、航班、取车、还车这类关键事项要有清楚的状态感。

必须避免：

- 普通博客攻略页。
- 纯 Markdown 阅读器。
- 企业后台风格。
- 小红书长图风格。
- 传统旅行社行程单风格。
- 过度商务、灰暗、冷淡的界面。

## 3. 设计关键词

关键词：

- Playful
- Rounded
- Soft
- Visual
- Checklist-driven
- Travel-game
- Calm but lively
- Mobile-first

中文解释：

- 活泼，但不能幼稚。
- 大圆角，但不能廉价。
- 信息多，但不能压迫。
- 图片丰富，但不能变成杂志海报。
- 动效明显，但不能影响阅读。
- 有游戏化，但不能真的做成游戏。

## 4. 页面结构

底部主导航建议保留 5 个入口：

1. 今日
2. 行程
3. 选择
4. 待办
5. 资料

### 今日

旅途中默认打开的页面。

应展示：

- 当前日期 / Day 编号。
- 今日城市或路线。
- 今日关键提醒。
- 今日时间线。
- 今日驾驶强度。
- 今日住宿。
- 今日必须完成的 To-do。
- 明天是否需要早起。

视觉要求：

- 顶部可以有大图或渐变图片遮罩，但不能变成营销 Hero。
- 核心内容必须第一屏可见。
- 时间线用大圆角时间块或路径节点表达。

### 行程

完整 Day 列表。

应展示：

- Day 1 到 Day 13。
- 日期、城市、路线摘要。
- 驾驶强度。
- 是否早起。
- 是否有预订事项。
- 是否为核心景点日。

视觉要求：

- 可参考 Duolingo 的路径感，但不要牺牲扫描效率。
- 每天可以是一个圆角节点卡。
- 已完成、待进行、重点日、缓冲日要有明显状态。

### Day 详情

每一天的完整路书页面。

应展示：

- 当天标题，例如 09.27 Day 4 Twizel、Mt Cook、Wanaka。
- 当天一句话摘要。
- 当天重点。
- 时间顺序安排。
- 地点卡片。
- 交通 / 驾驶信息。
- 住宿建议。
- To-do。
- 备注和风险提示。

视觉要求：

- 不照搬 Markdown 段落。
- 时间线是核心组件。
- 地点名称必须配中文类型和简短解释。
- 重要提醒使用醒目的提示组件，不用大段红字。

### 选择

给同行人做选择的页面。

适合放：

- Queenstown 活动选项。
- Onsen / Skyline / Glenorchy / 蒸汽船等可选项目。
- 悉尼中转路线 A / C。
- Twizel vs Tekapo 住宿选择。
- Crown Range vs Cromwell 路线选择。

视觉要求：

- 每个选项必须图文并茂。
- 选项卡要包含适合人群、耗时、费用、体力强度、优缺点。
- 支持明显的选中状态。
- 可使用 Duolingo 式大按钮和轻微弹性动效。

### 待办

出发前和旅途中所有需要确认的事项。

应按类别组织：

- 航班 / 跨境交通
- 签证 / 证件
- 住宿
- 租车
- 活动预订
- 行李 / 装备
- 当天临时确认

视觉要求：

- 参考 Finch 的 checklist 感。
- checkbox 要有完成动画。
- 任务卡要有优先级、截止时间、关联日期。
- 完成后可以轻微下沉或变淡，但不能消失。

### 资料

长期查阅的信息中心。

应包含：

- 航班信息。
- 租车和右舵驾驶注意事项。
- 签证和证件。
- 携带清单。
- 预算。
- 新西兰道路和天气注意事项。
- 紧急联系和常用链接。

视觉要求：

- 信息密度可以比其他页面高。
- 用分组卡片和图标入口，避免长文堆叠。

## 5. 核心组件规范

### Bottom Tab

底部导航必须是浮动式、大圆角、半透明或玻璃质感。

要求：

- 固定在底部。
- 有安全区适配。
- 选中态明显。
- 图标 + 短文字。
- 切换时有轻微滑动或缩放反馈。

不要做成普通网页页脚。

### Checklist Button

用于 To-do 和携带清单。

要求：

- 点击区域足够大。
- 未完成、完成、重要、逾期状态要可区分。
- 完成时有动画反馈，例如打勾弹入、卡片轻微缩放、颜色变浅。
- 文案要完整，不要只显示短标签导致信息不足。

### Tabs / Segmented Control

用于 Day 分类、选择页分类、资料页分类。

要求：

- 大圆角胶囊形。
- 有滑块式选中背景。
- 横向滚动时要顺滑。
- 文字不能挤压或溢出。

### Day Card

用于行程列表。

要求：

- 包含日期、Day、路线、当天性质、驾驶强度。
- 可以有小图或地点缩略图。
- 大圆角，轻阴影或柔和边框。
- 状态表达清楚：核心日、缓冲日、早起日、移动日。

### Timeline Item

用于 Day 详情。

要求：

- 左侧时间，右侧内容。
- 重要节点可以更大。
- 驾驶、景点、吃饭、入住、航班、活动要有不同图标或色块。
- 不能使用纯文本流水账。

### Place Card

用于地点和景点。

要求：

- 必须有图片，除非该地点本身只是机场、酒店、租车点这类功能点。
- 必须包含英文名、中文名、地点类型、简短解释。
- 对首次了解新西兰的人要能看懂这个地方是什么。
- 可以包含推荐停留时间、是否必去、天气依赖程度。

### Option Card

用于同行人讨论和选择。

要求：

- 大图优先。
- 标题清楚。
- 提供适合人群、耗时、费用、体力强度、优点、缺点。
- 选中态要强，有明确按钮。
- 多个选项之间要能快速比较。

### Alert / Note

用于风险提示。

类型：

- 必须确认
- 天气相关
- 驾驶风险
- 早起提醒
- 预订提醒
- 证件提醒

要求：

- 使用柔和但明确的色块。
- 不要大量红色。
- 不要打断主要阅读流。

## 6. 圆角和形状

整体必须使用大圆角语言。

建议：

- 页面容器：28px - 36px。
- 大卡片：24px - 32px。
- 普通卡片：20px - 28px。
- 小按钮 / 标签：999px 胶囊。
- 图片：24px - 32px。
- 底部导航：28px - 36px。

注意：

- 大圆角必须配合足够留白。
- 不要所有元素都一个圆角值。
- 小标签可以胶囊化，大内容卡不要过度像气泡。

## 7. 色彩方向

色彩要来自新西兰自然环境，但用 Duolingo / Finch 式的饱和度和亲和力处理。

建议色板：

- Lake Blue：湖水蓝，用于主按钮、路线、选中态。
- Moss Green：草地绿，用于完成、可行、轻松状态。
- Sun Yellow：阳光黄，用于提示、亮点、选择页。
- Glacier White：冰川白，用于背景和卡片。
- Slate Ink：深灰蓝，用于正文。
- Coral Orange：橙红，用于关键提醒，但少量使用。

禁止：

- 全页面单一蓝色。
- 大面积紫蓝渐变。
- 过多米色 / 咖色。
- 纯黑暗黑模式作为默认。
- 红色警告泛滥。

## 8. 字体和文字层级

使用系统字体优先：

- iOS/macOS: -apple-system, BlinkMacSystemFont, "SF Pro Text", "PingFang SC"
- 通用 fallback: "Microsoft YaHei", Arial, sans-serif

层级建议：

- 页面大标题：28px - 34px，粗体。
- Day 标题：24px - 30px，粗体。
- 卡片标题：18px - 22px，粗体。
- 正文：15px - 17px。
- 辅助说明：13px - 14px。
- 标签：12px - 13px。

规则：

- 不要用 viewport width 控制字号。
- 中文行高要舒服，正文建议 1.55 - 1.75。
- 英文地名后必须跟中文解释或类型，避免同行人看不懂。
- 按钮文字不能溢出。

## 9. 图片使用规则

图片是本项目的重要视觉资产，不是装饰。

要求：

- 主要景点必须配真实图片。
- 图片要能帮助同行人理解这个地方，而不是只做氛围。
- 图片必须清晰、明亮、主体明确。
- 同一页面图片比例要统一。
- 不要使用黑暗、模糊、过度裁切、纯氛围图。

图片组件建议：

- Day 顶部：宽图，16:9 或 4:3。
- Place Card：4:3 或 1:1。
- Option Card：大图 16:10。
- 小列表缩略图：1:1。

## 10. 动效规范

动效要让页面更有生命力，但不能拖慢查信息。

建议动效：

- 页面切换：轻微 slide + fade。
- 卡片进入：staggered fade up。
- Tab 切换：滑块移动。
- checklist 完成：scale 0.96 -> 1.00，勾选弹入。
- Day 节点：当前日轻微 breathing 或高亮。
- 图片加载：blur-up 或淡入。
- Option 选中：卡片抬起、边框高亮、按钮状态切换。

限制：

- 单个动效建议 180ms - 360ms。
- 页面切换不超过 450ms。
- 不做大面积视差滚动。
- 不做影响阅读的持续循环动画。
- 必须尊重 prefers-reduced-motion。

## 11. 内容呈现规则

内容来自路书，但必须结构化展示。

每个英文地点名后应补充：

- 中文名。
- 地点类型。
- 一句话解释。

示例：

- Lake Tekapo（蒂卡波湖 / 湖泊与小镇）：南岛中部著名蓝色冰川湖，适合湖边拍照和看好牧羊人教堂。
- Twizel（特威泽尔 / 中转小镇）：位于 Tekapo 和 Mt Cook 附近，住宿通常比景区更好订，适合作为自驾落脚点。
- Aoraki / Mt Cook National Park（库克山国家公园 / 雪山徒步区域）：新西兰最高峰所在区域，Hooker Valley Track 是首次南岛旅行非常值得安排的轻徒步。

禁止：

- 只写英文名不解释。
- 长段 Markdown 直接塞页面。
- 同一屏出现过多连续文字。
- 把 To-do 写成普通段落。

## 12. 手机端布局规则

本项目以手机端为主。

目标尺寸：

- iPhone 15 / 16 / 17 标准宽度。
- 兼容小屏 iPhone SE 宽度，但不以它作为主要审美目标。
- 桌面端可以居中显示手机宽度容器，或者使用双栏增强。

规则：

- 第一屏必须展示当前页面的核心信息。
- 底部导航不能遮挡内容。
- 横向滚动组件要有明确可滑动暗示。
- 所有点击目标建议不小于 44px。
- 图片和卡片必须有稳定尺寸，避免加载后布局跳动。

## 13. 需要避免的设计错误

绝对避免：

- 把整篇 Markdown 渲染成一个文章页。
- 大段文字堆叠。
- 每个页面都做成卡片套卡片。
- 卡片圆角小、阴影硬、像后台系统。
- 只用一种颜色做全站。
- 过度依赖图标而没有文字解释。
- 图片只是占位，没有真实信息价值。
- 日程和 To-do 混在一起。
- 选择页没有图片和比较维度。
- 动效过多导致查信息变慢。

## 14. 实现建议

数据层建议结构化：

- days：每日行程。
- timelineItems：每日时间线。
- places：地点资料。
- todos：待办事项。
- options：可选择方案。
- transport：航班、包车、租车、火车、Uber 等交通信息。
- assets：图片资源。

Markdown 只作为内容来源和人工校对参考，不作为最终 UI 渲染方式。

组件优先级：

1. App Shell + Bottom Tab。
2. Today Page。
3. Trip Days Page。
4. Day Detail Page。
5. Checklist / Todo Page。
6. Option Cards。
7. Place Cards。
8. Info Pages。

推荐依赖：

- React + TypeScript。
- Framer Motion 或 Motion One，用于高质量动效。
- Lucide React，用于图标。
- CSS variables，统一设计 token。

## 15. 验收清单

每次做完一个页面，都要检查：

- 是否像 2026 年的移动端 App，而不是网页文章。
- 是否有大圆角、柔和色彩和明确动效。
- 第一屏是否能看懂这个页面的核心作用。
- 所有地点是否有中文类型和解释。
- 行程是否按时间顺序清楚展示。
- To-do 是否真的可以被当作 checklist 使用。
- 选择页是否能帮助同行人做决定。
- 图片是否真实帮助理解地点。
- 底部导航是否像 App，不像网页菜单。
- 手机宽度下文字是否溢出。
- 是否避免了纯 Markdown 渲染。

如果某个实现违反这份文档，应优先修改实现，而不是修改文档。
